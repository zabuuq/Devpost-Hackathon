**What I need to play the game and take screenshots**

**1. Where to run the game**
- A live URL where the HTML5 export is hosted and reachable from a normal browser (itch.io, GitHub Pages, Netlify, your own domain, etc.). If it's gated behind a password or itch "draft" link, include the access details.
- Confirmation that the build is the version you want screenshotted, plus the date/version tag so we know what we're capturing.
- Any known load-time quirks (e.g., "wait ~20 seconds for WASM to finish loading," "click anywhere to start audio").

**2. Game identity and framing**
- Official title of the game.
- One-sentence elevator pitch (so I can caption screenshots if needed).
- Genre and perspective (2D platformer, top-down, point-and-click, turn-based, etc.) — this affects how I pace inputs.
- Target resolution / aspect ratio of the canvas, and whether fullscreen is recommended.

**3. Controls — complete reference**
- Every keyboard key, mouse button, and on-screen control, with the action it performs.
- Which key/click focuses the canvas so keyboard input starts working.
- Any modifier combos, hold-vs-tap distinctions, or context-sensitive controls.
- Pause, menu, and restart bindings.
- If there's gamepad-only functionality with no keyboard fallback, flag it now — I can't use a gamepad.

**4. The intended playthrough path**
- A step-by-step walkthrough of the specific path you want captured, from page load to end state. Be explicit: "press Enter at title → choose 'New Game' → walk right until you see the first enemy → jump on it → ..."
- Approximate time each step should take so I know when to wait vs. act.
- Any required choices at branching points (which dialogue option, which door, which character).
- The end state that means "done" (credits, a specific screen, a certain level cleared).

**5. Screenshot plan**
- The exact list of moments you want captured, in order. For each one: a short name, what should be on screen, and roughly when it occurs in the walkthrough.
- Whether you want full-page screenshots or just the game canvas region.
- Preferred filename pattern (e.g., `01_title.png`, `02_tutorial.png`).
- Target resolution for the screenshots if it matters.
- Where to save them — I'll put them in the outputs folder unless you say otherwise.

**6. How to Play section alignment**
- The final list of controls, mechanics, and features that the How to Play section will reference, so my screenshots match the written pitch. If the pitch highlights "combo system" or "dialogue wheel," I need to know to capture those specifically.

**7. Known issues and safe-mode instructions**
- Any bugs, soft-locks, or dead ends to avoid.
- A "safe" recovery action if I get stuck (e.g., "press Esc then R to restart the level").
- Any cheat codes or debug flags that skip to specific scenes — useful if a scene is hard to reach through normal play.

**8. Reaction-time warning**
- Flag any section that requires fast reflexes (tight platforming, bullet-hell, timed QTEs). I have input latency between screenshot and keypress, so I'll need either an easy mode, invincibility, a checkpoint past that section, or explicit permission to skip it.

**9. Audio and popups**
- Whether browser audio autoplay prompts or cookie banners appear on the hosting page, and how to dismiss them.
- Whether the game itself has modal popups I should expect and dismiss.

**10. Delivery**
- The file I should read once this is ready: filename and location in the outputs folder.
- Any deadline or order-of-operations constraints.
